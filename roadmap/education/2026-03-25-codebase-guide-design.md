# Codebase Guide — Design Spec

> Feature: Living codebase structure guide that explains what goes where and why
> Date: 2026-03-25
> Status: Draft

## Problem

Understanding *why* a codebase is organized the way it is — not just *what* files exist — is critical for navigating and debugging without hand-holding. This is true for both greenfield projects (where standard conventions apply) and existing codebases (where the structure has evolved organically). Today, this knowledge lives in developers' heads or nowhere.

## Solution

A Claude Code skill + hook system that generates and maintains a `.forge/codebase-guide.md` file. The guide explains every folder and key file in plain English — what it does, why it exists, and when you'd touch it.

### Design Principles

- **Zero friction** — auto-generates on first session, updates via hooks
- **Works anywhere** — standalone by default, enriched when FORGE scan data is available
- **Teaches, doesn't just document** — explains the "why" behind structure, not just the "what"
- **Living document** — grows with the project, never goes stale

## Output: `.forge/codebase-guide.md`

### Structure

```markdown
# Codebase Guide

> Auto-generated by FORGE. Last updated: {date}
> Depth: {depth_level} (configure in .forge/config.yml)

## Project Type
{framework} application ({language} {version})

## Why This Structure?
{1-2 paragraph explanation of the structural pattern this project follows
and why the framework/community settled on it}

## Folder Map

### `{folder}/`
{purpose}

#### `{folder}/{subfolder}/`
{purpose — what goes here and why it's separate from other folders}

## Key Files
| File | What it does | When you'd touch it |
|------|-------------|-------------------|
| {file} | {purpose} | {trigger — what problem or task leads you here} |

## Data Flow
{how requests/data moves through the system — which folders talk to which}

## Recent Changes
{appended by post-commit hook — structural changes only}
```

### Depth Levels

| Level | Target audience | Style |
|-------|----------------|-------|
| `beginner` | New to programming | Explains concepts inline ("An endpoint is like a URL your app responds to") |
| `standard` (default) | Knows code, new to project structure | Explains conventions and the "why" behind them |
| `advanced` | Experienced, wants quick reference | Terse, assumes familiarity with patterns |

## Skill: `/codebase-guide`

### Three Modes of Operation

#### 1. Auto-trigger (first session)

**Trigger mechanism:** The skill is registered in Claude Code's skill system. On session start, if the skill's `auto_generate` config is `true`, it checks for `.forge/codebase-guide.md`. If missing, it runs guide generation before any other work. This is a skill-level check, not a Git hook or system hook.

When Claude starts a session in a project and no `.forge/codebase-guide.md` exists:

1. Detect project type from manifests (`package.json`, `pyproject.toml`, `Cargo.toml`, etc.)
2. Walk the file tree (respecting `.gitignore`, skipping `node_modules`/`dist`/etc.)
3. Check for FORGE scan data at `{target_repo}/.artifacts/scan/codebase_map.json` (this file lives inside the scanned project, not the FORGE engine repo)
   - If exists: pull module purposes, data flows, entry points, auth boundaries, `architecture_summary`, and `key_patterns`
   - If not: use Claude Code's own context (no additional API cost) to infer folder purposes from names, file contents, and import patterns
4. Generate `.forge/codebase-guide.md`
5. Notify: "Generated codebase guide at `.forge/codebase-guide.md`"

#### 2. Greenfield (new project)

When Claude creates a new project structure:

1. Look up the framework in the **convention library** (`forge/guide/structures/`)
2. As Claude creates folders/files, the skill writes the guide explaining why each exists
3. No LLM call needed for known frameworks — purely deterministic from convention YAML

#### 3. Update (ongoing)

When Claude creates, moves, or deletes files during a session:

1. Check if the change is structural (new folder, new file in a new location, deleted module)
2. If yes: update the relevant section of the guide
3. Non-structural changes (editing existing files) are ignored

### Refresh Command

`/codebase-guide refresh` — regenerates the full guide from scratch. Useful when:
- The guide has accumulated many "Recent Changes" entries
- Major refactoring has happened
- Staleness warning has been triggered

## Convention Library

Deterministic knowledge base of standard project structures per framework.

### Location

`forge/guide/structures/`

> Note: The existing `forge/conventions/` module handles linting, testing, and code style conventions. Structure conventions are a separate concern and live in `forge/guide/` to avoid mixing responsibilities.

### Format

```yaml
framework: fastapi
language: python
structure:
  app/:
    purpose: "Main application package. All server code lives here."
    why: "Python requires a package directory. Keeps app code separate from config, tests, and tooling."
    touch_when: null
    children:
      main.py:
        purpose: "App entry point. Creates the FastAPI instance and registers routers."
        why: null
        touch_when: "Adding new route modules or middleware."
      routes/:
        purpose: "API endpoint definitions. One file per resource."
        why: "Separating routes by resource keeps files focused and navigable."
        children: {}
      models/:
        purpose: "Database table definitions (SQLAlchemy ORM)."
        why: "Models define your data shape. Separate from routes because persistence is a different concern."
        children: {}
      schemas/:
        purpose: "Request/response Pydantic models."
        why: "Your API's public interface shouldn't mirror your database 1:1."
        children: {}
      services/:
        purpose: "Business logic. Routes call these, these call models."
        why: "Keeps routes thin. Shared logic between routes lives here."
        children: {}
  tests/:
    purpose: "Mirrors app/ structure."
    why: "Predictable test location — tests/routes/test_users.py tests app/routes/users.py."
  alembic/:
    purpose: "Database migration scripts."
    why: "Versioned schema changes so any database can be brought up to date."
```

### Initial Coverage

- FastAPI
- Next.js (App Router)
- Django
- Express.js
- Flask
- Go (standard layout)

More added over time as needed.

### How It's Used

- **Greenfield:** Claude creates structure, skill looks up convention, writes guide from YAML (no LLM cost)
- **Existing:** if project matches a known framework, convention data enriches the LLM-generated guide with "why" explanations

## FORGE Enrichment

When `.artifacts/scan/codebase_map.json` exists from a FORGE scan, the guide gets additional sections.

### What FORGE Adds

| Section | Source Field | What It Explains |
|---------|-------------|-----------------|
| Data Flow | `data_flows` | How requests/data move through the system, which modules talk to which |
| Auth Boundaries | `auth_boundaries` | Which routes are protected, what auth type, where middleware lives |
| Entry Points | `entry_points` | All ways into the system (API, CLI, worker, cron) |
| Module Relationships | `modules` + dependencies | Which modules depend on which, which are leaves |
| Architecture Summary | `architecture_summary` | Feeds into the "Why This Structure?" section |
| Key Patterns | `key_patterns` | Notable patterns (auth middleware, error handling, etc.) |

### Merge Strategy

1. Skill generates base guide (tree walk + LLM or convention library)
2. Checks for `.artifacts/scan/codebase_map.json`
3. If exists: pulls data flows, auth boundaries, entry points, module relationships
4. Weaves them into relevant guide sections (data flow into "Data Flow" section, etc.)
5. Adds "Deep Analysis" subsection if insights don't fit existing sections

### Without FORGE Data

Guide works fine — structure + purpose without cross-cutting concerns. Quality scales with investment in scanning.

## Hooks

### Post-commit Hook (Claude Code Hook)

This is a **Claude Code hook** (configured in `.claude/settings.json`), not a Git hook. It runs within Claude Code sessions only — manual commits outside Claude are caught by the staleness check on next session start.

**Hook type:** `PostToolUse` on the `Bash` tool, filtering for `git commit` commands.

**Behavior:**

1. Extract file paths from the commit diff (`git diff-tree --no-commit-id --name-status -r HEAD`)
2. Check for structural changes:
   - New files or folders (status `A`)
   - Deleted files or folders (status `D`)
   - Renamed or moved files (status `R`)
3. If structural:
   - Append entry to "Recent Changes" section of the guide:
     ```markdown
     ### 2026-03-25 — feat(auth): add OAuth provider
     - Added `app/auth/providers/` — new folder for OAuth provider integrations
     - Added `app/auth/providers/google.py` — Google OAuth handler
     ```
   - Print terminal notification: `Structure update: added app/auth/providers/ (OAuth integrations)`
4. If not structural: no action

**Installation:** The skill automatically adds the hook to `.claude/settings.json` when first run. No manual setup needed.

### Staleness Detection

When "Recent Changes" accumulates 10+ entries, or a **major structural shift** occurs, the hook updates the guide header:

```markdown
> Warning: Guide may be stale — run `/codebase-guide refresh` to regenerate
```

**Major structural shift** is defined as:
- Any change to a directory that is a direct child of the repository root
- Deletion of a directory containing 5+ files
- Addition of a new top-level directory

On session start, if the guide exists but the last-modified date is older than the most recent commit that added/deleted files, the skill warns about potential staleness.

## Configuration

Stored in `.forge/config.yml` — a **project-local config file** in the target repo, completely separate from FORGE's `ForgeConfig` (which governs scan/evaluation pipeline settings). The skill implements its own lightweight YAML config loader.

```yaml
codebase_guide:
  depth: standard        # beginner | standard | advanced
  auto_generate: true    # generate on first session if missing
  exclude_patterns:      # folders/files to skip in the guide
    - "*.test.*"
    - "__pycache__"
    - ".git"
    - "node_modules"
```

**Defaults when file does not exist:** All values above are defaults. The skill works without any config file — it only reads `.forge/config.yml` if present.

**Version control:** `.forge/codebase-guide.md` should be committed by default (team knowledge sharing). Users can add `.forge/` to `.gitignore` if they prefer local-only guides.

## Scope Boundaries

### In scope
- Guide generation (greenfield + existing)
- Convention library (YAML per framework)
- Post-commit hook (structural change detection + notification)
- FORGE enrichment (when scan data exists)
- Configurable depth levels
- `/codebase-guide` and `/codebase-guide refresh` skill commands

### Out of scope
- IDE integration (VS Code panel, etc.) — future enhancement
- Full monorepo support (per-package guides) — future enhancement. For now, monorepo detection (`workspaces` in `package.json`, `pnpm-workspace.yaml`) produces a warning: "Monorepo detected — guide covers root structure only."
- Guide diffing between versions — future enhancement
- Automatic framework migration detection — future enhancement

## Graceful Degradation

| Scenario | Behavior |
|----------|----------|
| No manifest files | Generic guide with folder listing and best-effort purpose inference |
| Large tree (10,000+ files) | Cap at 4 directories deep, note that deeper levels are omitted |
| LLM call fails/times out | Convention-only or tree-structure-only guide with a note that purposes could not be inferred |
| Unknown framework | Skip convention enrichment, use LLM inference or produce structure-only output |
| No FORGE scan data | Base guide only (no data flow, auth boundaries, etc.) — still useful |

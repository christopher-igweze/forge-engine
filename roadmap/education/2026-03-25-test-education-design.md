# Test Education & Audit — Design Spec

> Feature: Living test quality guide with reward-hacking detection and plain-English test explanations
> Date: 2026-03-25
> Status: Draft

## Problem

AI coding assistants write tests that pass but may not protect anything meaningful. Common reward-hacking patterns include: weak assertions that never fail, happy-path-only coverage, implementation-coupled tests that break on refactor, and cosmetic tests that inflate count without adding real coverage. The developer reviewing these tests often can't tell the difference — they see green checkmarks and move on.

Beyond reward-hacking, there's a knowledge gap: understanding what features exist, what tests *should* cover them, and whether existing tests would actually catch real bugs.

## Solution

A two-layer analysis system: deterministic pattern matching in the FORGE scan pipeline (cheap, catches lazy patterns) plus LLM-powered semantic analysis as a Claude Code skill (catches subtle reward-hacking via breakage reasoning). Produces a living `.forge/test-guide.md` that maps features to tests, explains each test in plain English, and suggests improvements.

### Design Principles

- **Breakage-first reasoning** — the core question is "would this test fail if the feature broke?", not "does this test look correct?"
- **Education, not just detection** — explains *why* a test is weak and shows what a strong test looks like
- **Two layers** — cheap deterministic checks for obvious patterns, LLM for semantic analysis
- **Feedback loop for AI assistants** — verdicts and suggestions are structured so your coding AI can act on them directly

### Verdict Definitions

| Verdict | Meaning |
|---------|---------|
| **Strong** | Would catch real breakage in the feature. Asserts on behavior and output, not implementation. |
| **Weak** | Looks correct structurally but wouldn't catch likely bugs. Explanation of *why* provided. |
| **Cosmetic** | Passes but protects nothing meaningful. Could be deleted with no loss of safety. |

### Layered Detail

Each test has two levels of explanation:
- **One-liner** (default): "Checks that expired tokens are rejected at /dashboard"
- **Full breakdown** (expandable): "Creates a user, generates a token with expiry in the past, sends GET /dashboard with that token in the Authorization header, verifies 401 response with 'token expired' message. Matters because without this, expired sessions could access protected data."

The guide presents one-liners in the table. Full breakdowns appear below the table for tests flagged as Weak or Cosmetic.

## Layer 1: Deterministic Pattern Matching (FORGE Scan Pipeline)

Cheap, fast, zero LLM cost. Catches lazy reward-hacking patterns via AST analysis. Runs during every FORGE scan as part of the `test_quality` dimension.

### New Checks

Added to `forge/evaluation/checks/test_quality.py` alongside existing TST-001 through TST-007:

| Check ID | Name | Severity | Deduction | What It Catches |
|----------|------|----------|-----------|----------------|
| **TST-008** | Weak assertions | medium | -5 to -15 | `assert True`, `assert x is not None`, assertions that don't verify actual values. Detected by matching assertion nodes in the AST against a list of known-weak patterns. |
| **TST-009** | Mock overload | medium | -5 to -10 | More `@patch`/`mock` decorators than assertion statements in a test function. Indicates the test is testing mocks, not real code. Detected by counting mock decorators vs assertion nodes. |
| **TST-010** | Duplicate test bodies | low | -3 to -8 | Multiple test functions with identical or near-identical AST structure (>90% node similarity). Inflates test count without adding real coverage. Detected by AST hashing and comparison. |

### Detection Method

- Tree-sitter AST parsing (reuses `forge/graph/builder.py` infrastructure)
- Pattern matching on assertion nodes, mock/patch decorators, function bodies
- AST hashing for duplicate detection (normalize variable names, compare structure)

### Scoring

Each failed check applies a deduction to the `test_quality` dimension score (same mechanism as TST-001 through TST-007). Deduction scales with the number of violations found.

### Limitations

Layer 1 catches the *obvious* patterns. It cannot determine whether a structurally-correct test is semantically meaningful. That requires Layer 2.

## Layer 2: Semantic Analysis (Skill — LLM-Powered)

The real analysis. Uses Claude's own context (no additional API cost beyond the skill invocation) to reason about whether tests would catch actual breakage.

### Breakage Reasoning Process

For each feature:

1. **Read source code** — the feature module (e.g., `app/auth/login.py`)
2. **Read tests** — the test files mapped to that feature
3. **Enumerate breakage scenarios** — "What are the ways this feature could break?"
   - Removed validation logic
   - Bypassed auth checks
   - Wrong return values
   - Missing error handling
   - Race conditions / ordering issues
4. **Test each scenario against existing tests** — "If I applied this breakage, which tests would fail?"
5. **Produce verdicts:**
   - **Strong** — test would catch this breakage
   - **Weak** — test runs the code path but doesn't assert on the right thing (explain why)
   - **Cosmetic** — test would pass even with the breakage applied
6. **Identify uncovered scenarios** — breakages that no existing test would catch
7. **Generate suggested tests** — real, runnable code for the uncovered scenarios

### Example Output

```
test_login_success
  Summary: Checks that valid credentials return a 200 with a JWT
  Verdict: Strong
  Protects against: removed auth logic, broken token generation

test_token_valid
  Summary: Checks that a token is not None
  Verdict: Cosmetic — would pass even if the token contained garbage data
  Breakage it misses: malformed tokens, wrong user ID in payload, missing expiry
  Better version:
    def test_token_contains_correct_claims(client, test_user):
        """Token should contain user ID, role, and valid expiry."""
        response = client.post("/login", json=valid_credentials)
        token = response.json()["token"]
        payload = jwt.decode(token, SECRET, algorithms=["HS256"])
        assert payload["user_id"] == test_user.id
        assert payload["role"] == "user"
        assert payload["exp"] > time.time()
```

### Feature Detection

Features are inferred from code using these sources (in order of preference):

1. **Codebase guide** — if `.forge/codebase-guide.md` exists, its module descriptions serve as the feature list
2. **Routes/endpoints** — API endpoint definitions (each route group = a feature)
3. **Services** — business logic modules
4. **Models** — data model definitions (each model with associated logic = a feature)
5. **CLI commands** — if the project has a CLI

### Test-to-Feature Mapping

Tests are matched to features by:
1. **Naming convention** — `test_auth.py` maps to `auth/`
2. **Import analysis** — `from app.auth import login` in a test links it to the auth feature
3. **File proximity** — `tests/auth/` maps to `app/auth/`
4. **Fallback** — unmapped tests are listed in an "Unmapped Tests" section

## Skill: `/test-audit`

### Modes of Operation

#### Generate (first run or on demand)

When no `.forge/test-guide.md` exists or you run `/test-audit`:

1. Detect features (from codebase guide, routes, services, models)
2. Map test files to features (naming, imports, proximity)
3. Run Layer 2 semantic analysis for each feature
4. If `.artifacts/` exists from a FORGE scan, merge Layer 1 deterministic results
5. Generate `.forge/test-guide.md`
6. Notify: "Generated test guide at `.forge/test-guide.md`"

#### Real-time Feedback (Post-commit Hook)

Claude Code hook — `PostToolUse` on `Bash`, filtering for `git commit` commands.

**Behavior:**

1. Check if committed files include test files (match `test_*`, `*_test.*`, `*.spec.*`, files in `tests/` or `__tests__/`)
2. If yes: run Layer 2 analysis on *only the changed tests* and their corresponding source modules
3. Print a short verdict:
   ```
   Test audit: 3 new tests in tests/auth/test_oauth.py
     Strong: test_oauth_redirect — protects against broken redirect URI
     Weak: test_oauth_callback — doesn't verify token exchange, only checks 200
     Cosmetic: test_oauth_logout — asserts response is not None

   Suggestion: test_oauth_callback should verify the access token contains correct scopes
   ```
4. Append summary to "Recent Analysis" section of test guide

**Scope:** Only analyzes changed test files per commit, not the full codebase. Full analysis requires `/test-audit refresh`.

**Installation:** Skill automatically adds the hook to `.claude/settings.json` when first run.

#### Refresh

`/test-audit refresh` — regenerates the full guide from scratch.

### Connection to Codebase Guide

When both `.forge/codebase-guide.md` and `.forge/test-guide.md` exist:

- Codebase guide's folder map links to test guide's feature sections
- Test guide's feature-to-test map uses codebase guide's module descriptions for feature names and purposes
- Both guides update independently via their respective hooks
- `/test-audit` uses module purposes from the codebase guide to write more contextual breakage scenarios (e.g., "This module handles payment processing, so breakage scenarios focus on charge amounts, idempotency, and refund logic")

## Configuration

Added to `.forge/config.yml` (same file as codebase guide config):

```yaml
test_audit:
  auto_generate: true          # generate on first session if missing
  realtime_feedback: true      # post-commit analysis of new tests
  exclude_patterns:            # test files to skip
    - "tests/fixtures/*"
    - "tests/conftest.py"
    - "**/test_helpers.*"
  feature_sources:             # where to infer features from (order of preference)
    - codebase_guide           # .forge/codebase-guide.md if it exists
    - routes                   # API endpoint definitions
    - services                 # business logic modules
    - models                   # data models
```

**Defaults when file does not exist:** All values above are defaults. Works without any config file.

## Integration with Existing FORGE

| Component | Layer | Role |
|-----------|-------|------|
| TST-001 to TST-007 (existing) | 1 | Structural checks — missing tests, empty functions, low coverage, no config |
| TST-008 to TST-010 (new) | 1 | Reward-hacking pattern detection — weak assertions, mock overload, duplicates |
| `/test-audit` skill | 2 | Semantic analysis — breakage reasoning, plain-English explanations, suggestions |
| `.forge/test-guide.md` | Output | Standing report — feature map, verdicts, gaps, suggested tests |
| Post-commit hook | Real-time | Immediate feedback on newly written/modified tests |

Layer 1 results feed into Layer 2: when FORGE scan data exists at `{target_repo}/.artifacts/`, the skill incorporates deterministic check results so the guide shows both structural issues and semantic verdicts.

## Scope Boundaries

### In scope
- Test guide generation (`.forge/test-guide.md`)
- Layer 1 deterministic checks (TST-008 to TST-010) in FORGE scan pipeline
- Layer 2 semantic breakage analysis via `/test-audit` skill
- Feature-to-test mapping (inferred from code + codebase guide)
- Real-time post-commit feedback on changed test files
- Plain-English test explanations (layered: one-liner + full breakdown)
- Concrete test suggestions for gaps and weak tests
- Integration with codebase guide feature detection

### Out of scope
- Actual mutation testing (running mutated code against tests) — future enhancement
- Test execution or coverage measurement (FORGE analyzes test code, doesn't run tests)
- Auto-fixing weak tests (suggests improvements, doesn't apply them)
- Flaky test detection (requires execution history, not static analysis)
- Performance/speed analysis of tests — future enhancement

## Graceful Degradation

| Scenario | Behavior |
|----------|----------|
| No test files found | Guide shows "No tests detected" with a starter checklist of what to test first based on detected features |
| No features detected | Falls back to listing test files with explanations, without feature grouping |
| No codebase guide | Feature detection uses routes/services/models directly (less context, still works) |
| No FORGE scan data | Layer 1 results omitted, Layer 2 semantic analysis still runs |
| LLM analysis fails | Guide shows test-to-feature mapping with structural info only, notes that semantic analysis is unavailable |
| Very large test suite (500+ test files) | Prioritizes critical features first (auth, payments, data handling), notes remaining features as "not yet analyzed" |
